# Tamanu PostgreSQL tuning (Windows)

The `bestool tamanu pg-tune` command applies PostgreSQL performance tuning to a Windows server hosting Tamanu.
On Linux this tuning is applied by external configuration management; this command provides the equivalent for Windows, where tuning is otherwise manual.
The command locates the PostgreSQL install, computes tuning values from the host's resources, writes them to the server configuration, and offers to restart PostgreSQL and then the Tamanu workloads.
It is available only on Windows.

## Discovery

The command finds the PostgreSQL data directory and its major version from the standard Windows install layout, honouring explicit data-directory and version overrides when given.
The configuration file tuned is the `postgresql.conf` within that data directory.

## Tuning values

The tuning targets replicate the values produced by pgtune for an OLTP workload on SSD storage, with the resource budget and platform adjustments described below.
All memory quantities are computed in kibibytes to avoid unit drift, then rendered with the largest exact unit.

### Resource budget

PostgreSQL on a Windows host is treated as entitled to at most 4 GiB of RAM, because these hosts co-locate the application, reverse proxy, and backup tooling.
The memory budget is the smaller of half the host's physical RAM and 4 GiB.
For example, a 4 GiB host yields a 2 GiB budget, an 8 GiB host yields 4 GiB, and any host of 16 GiB or more yields 4 GiB.

The CPU budget reserves cores for the co-located workloads: 1 core maps to 1, 2 to 4 cores map to 2, 5 to 7 cores map to 4, and 8 or more cores map to the core count minus 4.

Every value below is derived from these budgeted figures, not the host's raw resources.

### Memory settings

- `shared_buffers` is a quarter of the memory budget, capped at 512 MiB on PostgreSQL versions before 10.
- `effective_cache_size` is three quarters of the memory budget.
- `maintenance_work_mem` is a sixteenth of the memory budget, capped at 2 GiB less 1 MiB on PostgreSQL 17 and earlier and at 8 GiB thereafter.
- `work_mem` is the memory budget minus `shared_buffers`, divided by three times the sum of the maximum connections and the parallelism allowance, with a floor of 4 MiB and, on PostgreSQL 17 and earlier, a ceiling of 2 GiB minus 1 MiB.
  The parallelism allowance is the CPU budget when parallelism is enabled, otherwise 8.
- `wal_buffers` is 3% of `shared_buffers`, with a floor of 32 KiB and a ceiling of 16 MiB; a computed value between 14 MiB and 16 MiB is raised to 16 MiB.
- `huge_pages` is `try` when `shared_buffers` is at least 2 GiB, otherwise `off`.

### Fixed settings

- `max_connections` defaults to 100 and is configurable.
- `checkpoint_completion_target` is 0.9.
- `default_statistics_target` is 100.
- `random_page_cost` is 1.1.
- `min_wal_size` is 2 GiB and `max_wal_size` is 8 GiB.
- `jit` is `off` on PostgreSQL 12 and later.

### Parallelism settings

Parallelism is enabled when the CPU budget is at least 4.
When enabled:

- `max_worker_processes` and `max_parallel_workers` are the CPU budget.
- `max_parallel_workers_per_gather` and `max_parallel_maintenance_workers` are half the CPU budget rounded up, capped at 4.

When the CPU budget is below 4, none of these are set.

`autovacuum_max_workers` is 5 for 32 or more cores and 4 for 16 or more, and is otherwise left at the server default.
`autovacuum_work_mem` is 2 GiB when `maintenance_work_mem` reaches 2 GiB, and is otherwise left at the server default.
On PostgreSQL 18 and later, `io_workers` is a quarter of the CPU budget clamped between 3 and 32, and is set only when that exceeds 3.

### Platform omissions

`effective_io_concurrency` is not set on Windows.
No WAL input/output method is forced on Windows.

### WAL compression

`wal_compression` is set to `lz4` on PostgreSQL 15 and later when the running server reports lz4 as a supported method, and to `on` otherwise.
On PostgreSQL 10 through 14 it is set to `on`.
When the server cannot be reached to determine supported methods, `on` is used, as it is always available.

### Temporary file limit

`temp_file_limit` is a quarter of the total size of the volume backing the data directory.
The proportion defaults to 25% and is configurable.

## Configuration block

The tuning is written as a single delimited block appended at the end of `postgresql.conf`, so that its settings override any earlier occurrences.
The block is bounded by a begin marker and an end marker, and the begin marker records that the block is generated by this command and must not be edited by hand.
Re-running the command replaces the content between the markers and leaves the rest of the file untouched, so repeated runs with unchanged inputs produce an identical file.

A server previously tuned by hand carries a bare header comment instead of the paired markers; the command recognises such a legacy header and replaces everything from it to the end of the file with the delimited block.

## Restart

After writing, the command compares the new block against what was already present.
When they are identical the server is already tuned and no restart is offered.
When they differ a PostgreSQL restart is required for the changes to take effect.

The command prompts separately to restart PostgreSQL and then to restart the Tamanu workloads.
Each prompt defaults to yes on a non-production host and to no on a production host.
A host is production when its Tailscale device name contains `prod`; when this cannot be determined the host is treated as production.
Non-interactive invocation can decline both restarts or accept both without prompting.

Restarting PostgreSQL stops and starts the PostgreSQL service for the tuned version.
Restarting the Tamanu workloads restarts the Tamanu processes under the process supervisor.

## Relationship to health checking

The Tamanu doctor's PostgreSQL tuning check and this command share one definition of the resource budget and expected values, so that a host tuned by this command passes the check.
On Windows the check evaluates its expectations against the same budget this command uses.
